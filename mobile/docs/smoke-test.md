# 실기기 스모크 테스트 (출시 전)

이 문서는 출시 전 8개 수정을 **실제 안드로이드 폰**에서 손으로 확인하는 절차다.
자동 단위/E2E 테스트(183개)가 WebView 로직은 커버하지만, SMS 발송·권한·로그인 자동입력 같은
**네이티브 흐름은 실기기에서만** 검증된다.

> 이 세션에는 폰이 연결돼 있지 않아(`adb devices` 비어 있음) 자동 구동은 못 했다.
> 아래는 폰을 USB로 연결한 뒤 직접 실행하는 절차다.

## 0. 준비

```bash
# 프로젝트 루트
cd mobile

# (최신 웹 반영 후) APK 빌드
npx cap sync android
cd android && ./gradlew :app:assembleDebug
# 산출물: app/build/outputs/apk/debug/app-debug.apk

# 폰 연결 확인 (USB 디버깅 ON)
adb devices          # 기기가 목록에 떠야 함

# 설치 (기존 데이터 유지하며 갱신: -r, 서명 다르면 먼저 삭제)
adb install -r app/build/outputs/apk/debug/app-debug.apk

# 로그 관찰 (별도 터미널) — 크래시/네이티브 로그 확인용
adb logcat -c && adb logcat | grep -iE "poncle|sms|AndroidRuntime|Capacitor"
```

테스트용 SMS는 **본인 번호**로 보내라(실제 고객에게 가지 않도록). 폰클 계정은 실제 계정 사용.

---

## 1. #1 발송 팬텀 방지 (전송 확인 후 기록) — **P0, 핵심**

가장 중요한 수정. "미발송인데 발송됨으로 남는" 조용한 누락을 막았는지 확인.

1. 설정 → **실제 발송 ON**. SMS 권한 허용.
2. 로그인 → 스캔 → 발송 대상 1명(본인 번호 테스트 대상 만들거나 실제 대상 중 본인 번호로 임시 변경).
3. **정상 발송:** 대상 카드 → `알림 보내기` → 확인 모달 → `보내기`.
   - ✅ "문자 보내는 중…" 토스트가 먼저 뜨고, **문자 실제 수신 후** "문자를 보냈습니다"로 바뀐다.
   - ✅ 이력 화면에 발송 기록이 남는다.
4. **전송 실패 재현(비행기모드):** 비행기모드 ON → 발송 시도.
   - ✅ "발송 실패" 토스트, 대상은 **미발송 상태 그대로**(카드에 `알림 보내기` 버튼 유지).
   - ✅ 이력에 그 고객 기록이 **생기지 않는다**(팬텀 없음).
   - ✅ 비행기모드 OFF 후 다시 스캔 → 그 고객이 여전히 발송 대상으로 뜬다(누락 안 됨).
5. **발송 직후 강제 종료:** `보내기` 누르고 문자 나가는 즉시 앱을 스와이프로 강제 종료 → 재실행.
   - ✅ 실제로 문자가 나갔으면 이력에 있고, 안 나갔으면 없음. "안 나갔는데 발송됨"은 없어야 한다.

## 2. #3 스캔 재진입 가드 — P1

1. 홈에서 `지금 다시 스캔` 누르고 **스피너 도는 중** 하단 탭으로 대시보드 이동 → `지금 스캔` 연타.
   - ✅ 스캔이 **중복 실행되지 않는다**(결과가 깜빡이거나 뒤바뀌지 않음). 진행 중이면 두 번째는 무시.
2. 스캔 완료 후 다시 누르면 정상 재스캔.

## 3. #5 SMS 권한 거부 → 설정 이동 안내 — P2

1. 안드로이드 설정에서 이 앱의 SMS 권한을 **거부(다시 묻지 않음)**로 만든다.
2. 앱에서 설정 → 실제 발송 토글 ON, 또는 발송 시도.
   - ✅ "문자 발송 권한이 꺼져 있습니다. 설정 화면에서 SMS 권한을 켜시겠어요?" 확인창.
   - ✅ `확인` → 이 앱의 **시스템 설정 상세 화면**이 열린다.
3. 설정에서 권한 켜고 돌아와 발송 → 정상.

## 4. #6 비밀번호 변경 후 옛 비번 자동입력 차단 — P3

1. 설정에서 폰클 자격증명(아이디/비번) 저장.
2. 폰클 웹에서 **비밀번호를 변경**(또는 저장된 비번을 일부러 틀리게 저장).
3. 앱에서 재로그인 → 로그인 WebView가 뜨고 저장된 (옛) 비번이 자동입력됨 → 그대로 로그인 시도(실패).
   - ✅ 1회 실패 후, 다시 로그인 화면 로드 시 **비밀번호 칸은 자동입력되지 않는다**(아이디는 채워짐).
   - ✅ "저장된 비밀번호가 맞지 않을 수 있습니다…" 안내 1회 노출.
4. 비번을 직접 올바르게 입력 → 로그인 성공(정상 흐름 안 깨짐).
5. 최초 자동입력(맞는 비번일 때 한 번에 로그인)이 여전히 잘 되는지도 확인.

## 5. #2 백업 복원 확인창 — P0

1. 발송 이력을 몇 건 쌓는다.
2. 설정 → 백업 내보내기(정상 백업 1개 확보).
3. 설정 → **백업 가져오기**로 이력이 더 적은/빈 파일 선택(예: 방금 설치 직후 만든 빈 백업).
   - ✅ "현재 발송 이력 N건이 백업의 M건으로 덮어써집니다 … 계속할까요?" 확인창.
   - ✅ `취소` → 이력 **그대로 유지**.
   - ✅ 정상(같거나 많은) 백업 복원 시엔 확인창 없이 바로 복원.

## 6. #8 백업 XSS 방어 — P3(단일 기기 전제상 낮음)

1. 텍스트 편집기로 백업 JSON을 열어 한 레코드의 `"channel"` 값을 `"<img src=x onerror=alert(1)>"`로 바꿔 저장.
2. 그 파일을 가져오기 → 이력 화면 열기.
   - ✅ 경고창(alert) **안 뜨고**, 태그 자리에 텍스트가 그대로(이스케이프되어) 보인다. 스크립트 실행 없음.

## 7. #7 스캔 오류 메시지 — P3

1. 로그인 안 된 상태나 비정상 응답을 유발(예: 세션 만료 상태에서 스캔).
   - ✅ 화면에 `TypeError…` 같은 영문 스택이 아니라 "스캔 중 오류가 발생했습니다. 다시 시도해 주세요." 류 한글 메시지만 보인다.
   - (상세 원인은 `adb logcat`의 `console.error`에 남는다.)

## 8. #4 전화번호 포맷 무관 발송 판정 — P1

폰클 두 엔드포인트(미결/개통)가 같은 번호를 다른 포맷(하이픈 유무)으로 줄 때 중복/누락이 없는지.
자동 테스트로 검증됨(하이픈↔맨숫자 매칭). 실기기에선 부수 확인만:

1. 한 고객에게 발송 → 이력 기록.
2. 재스캔 → 그 고객이 **이미 발송(✓)**으로 뜨고, 미발송으로 다시 뜨지 않는다.

---

## 통과 기준

- #1, #2 (P0): **반드시 통과**해야 출시.
- #3, #4 (P1): 통과 확인.
- #5, #6, #7, #8: 통과 확인(단일 기기 전제상 낮은 위험이나 회귀 없음 확인).
- `adb logcat`에 `AndroidRuntime`(크래시) 없어야 함.
